# Limitations and Future Work

## 1. 项目现状

“欧派AI灵感家——基于成交案例库的客户需求洞察与全案推荐系统”目前已经完成MVP原型，跑通了以下核心链路：

> 创建或切换匿名客户 → 输入完整基础需求 → 视觉偏好选择 → 实时生成客户画像 → 识别需求矛盾 → 推荐三套家居方案 → 解释推荐理由 → 展示交付进度与风险预警

当前版本已经能够验证“利用视觉选择降低客户需求表达难度，并通过结构化画像完成案例推荐”的产品思路。但系统仍主要用于比赛演示和方案验证，与可在真实门店和企业业务中部署的生产级系统之间存在明显差距。

现阶段更准确的产品定位是：

> 带有AI产品设计思路的视觉需求选择与规则推荐系统，而不是已经完整落地的AI家装顾问Agent。

### 1.1 本次更新后已经具备的能力

本次版本已经完成“完整需求输入与多客户模式”，主要新增和完善了以下能力：

- 按MVP产品需求文档采集匿名客户名、城市、户型、面积、预算、家庭结构、入住时间、特殊需求和自然语言描述；
- 支持创建、编辑、删除、切换和继续服务多名匿名客户；
- 保留“林女士”作为可随时恢复的一键演示客户；
- 视觉选择记录、客户画像、推荐预算、选中方案和交付风险处置均与当前客户绑定；
- 不同客户的数据在同一浏览器中相互隔离；
- 使用浏览器 `localStorage` 自动保存，刷新页面后可以恢复客户及其进度；
- 比赛版不采集手机号、身份证和详细地址；
- 项目根目录提供统一启动入口，评委从GitHub获取文件后可通过 `npm run setup`、`npm run dev` 和 `http://localhost:3000` 进行本地体验；
- 本地演示不依赖PostgreSQL、Express API或外部业务系统。

这些能力解决了原先固定使用“林女士”的演示痕迹，使系统能够在单台电脑、单个浏览器中为多名匿名客户分别完成演示流程。但这仍属于比赛版的本地多客户模式，不等同于已经实现企业级客户管理系统。

## 2. 当前项目的主要不足

### 2.1 案例数据以模拟数据为主

系统目前使用60条脱敏模拟案例，其中部分案例参考欧派官网公开信息，部分案例、业务属性和统计指标由AI生成。虽然数据覆盖了不同风格、户型、预算和家庭结构，但仍存在以下局限：

- 缺少欧派真实成交订单和真实客户反馈；
- 成交率、满意度、修改次数等指标属于模拟值；
- 无法反映不同城市、门店、设计师和客户群体之间的真实差异；
- 案例尚未与实际生产、库存、价格和交期建立关联；
- 当前数据规模不足以支持可靠的机器学习训练和效果评估。

因此，目前的推荐结果只能用于验证产品逻辑，不能直接作为真实报价、生产或商业决策依据。

### 2.2 推荐算法仍以规则评分为主

现阶段推荐算法主要根据风格、户型、预算、家庭结构、材质、灯光、收纳和生活需求等标签进行加权匹配。这种方式具有可解释、可控制的优点，但仍存在以下不足：

- 推荐权重主要依据产品假设和业务经验设定，尚未通过真实成交数据校准；
- 难以充分理解“温馨一点”“高级但不要太冷”等复杂、模糊的自然语言；
- 对客户隐性偏好和偏好变化的识别能力有限；
- 尚未形成“浏览—收藏—修改—成交—交付—评价”的学习闭环；
- 新风格、新产品和新客群进入系统后仍需要人工配置标签与规则。

### 2.3 视觉偏好选择仍采用固定轮次

系统已经支持整体氛围、装修风格、冷暖色、材质、收纳方式、灯光和个性元素七轮选择，但不同客户看到的交互路径基本一致。

实际业务中，不同客户需要不同的提问策略。例如，已经明确喜欢现代原木风的客户不需要重复完成大范围风格筛选；预算敏感客户应更早确认预算与必要功能；有老人、儿童或宠物的家庭则应优先确认安全、清洁和生活方式需求。

当前系统尚未实现根据客户回答、画像完整度和偏好置信度动态调整提问顺序、图片内容和选择轮次。

### 2.4 客户画像仍不够深入

当前画像主要记录风格、配色、材料、预算、家庭结构和收纳方式等显性需求，对真实生活方式的理解仍然不足，例如：

- 家庭成员的日常活动路线；
- 衣物、书籍、玩具和家电的具体数量；
- 做饭频率、厨房使用习惯和家庭用餐方式；
- 未来家庭成员变化和空间成长性；
- 居住者身高及无障碍需求；
- 采光、通风、噪声和朝向；
- 原有家具及家电保留情况；
- 客户对预算、效果、环保、收纳和交期的优先级排序。

因此，目前的画像更接近“装修偏好画像”，尚未完全达到“家庭生活方式画像”。

### 2.5 尚未真正实现可与客户对话的Agent

当前系统虽然使用了“AI家装顾问”的产品表达，但需求输入、视觉选择、矛盾识别和推荐说明主要由前端预设流程、结构化选项、规则算法和文字模板驱动，尚未接入能够持续与客户进行多轮自然语言沟通的AI Agent。

当前系统还不能真正做到：

- 根据客户回答主动追问；
- 理解上下文、省略表达和口语化需求；
- 记住客户前后多轮对话及画像修改记录；
- 判断哪些信息已经明确、哪些信息仍然缺失；
- 根据不同客户动态调整提问顺序；
- 通过自然语言修改局部画像；
- 解释预算、材料、收纳、安全和交付之间的取舍；
- 主动调用案例检索、推荐、报价和交付查询工具；
- 在无法确定或涉及高风险决策时转交设计师或销售人员。

例如，当客户表示“想要温馨一点，但不要像奶油风那么白，家里还有小孩，东西比较多”时，真正的Agent应能够结合上下文继续询问木色深浅、收纳比例和儿童安全要求，并在必要时调用视觉选择器展示对比图片。当前系统尚不能完成这种自然、连续、因人而异的交流。

### 2.6 图片与真实空间条件结合不足

当前图片主要用于表达风格和视觉偏好，系统尚未接入真实户型图、现场量尺和空间结构数据，可能出现以下问题：

- 客户喜欢的方案不适合其实际户型；
- 效果图空间比例与真实房屋不一致；
- 未充分考虑梁柱、门窗、水电和承重结构；
- 图片中展示的柜体尺寸无法直接用于生产；
- 效果图与最终交付结果之间仍可能存在差异。

当前系统主要解决“客户喜欢什么”，尚未完全解决“客户的房子能否这样设计和交付”。

### 2.7 真实SKU、报价与供应链数据尚未接入

系统可以模拟展示SKU、预算和交付风险，但尚未连接欧派内部业务系统，因此无法实时获得：

- 产品是否在售及所在地区是否可售；
- 门店实际价格、折扣和促销政策；
- 板材、五金和配件库存；
- 工厂生产周期和跨品类齐套情况；
- 物流、安装人员和服务资源；
- 正式报价单、合同及订单状态。

因此，当前预算、产品和交期信息只能用于演示，不能直接成为正式报价或客户承诺。

### 2.8 交付看板仍由模拟事件驱动

当前交付看板覆盖量尺、设计、报价、下单、生产、物流、安装和验收，并包含报价漏项、产品延期和跨品类交期不一致等风险场景，但进度和风险主要由模拟数据触发。

现阶段系统还不能：

- 实时反映真实订单状态；
- 根据历史订单预测延期概率；
- 自动确定风险责任人和处理时限；
- 执行预警、处理、复核、升级和关闭的完整闭环；
- 根据客户承诺和实际进度自动生成通知。

### 2.9 AI效果图尚未形成稳定生成链路

当前系统使用30张自有演示图片支撑视觉选择和案例推荐，并在前端明确标注“AI生成示意图”。这种方式适合MVP展示，但仍存在以下不足：

- 尚未根据每位客户的画像实时生成个性化效果图；
- 无法保证多个空间之间的风格、材质和产品完全一致；
- 图片中的部分结构可能无法真实生产；
- 无法保证所有元素都对应欧派真实在售SKU；
- 局部修改能力有限；
- 效果图尚未与报价、CAD、工艺和生产文件联动。

### 2.10 用户测试和量化验证不足

现有交互流程和业务指标主要基于产品设计假设，尚缺少足够规模的真实用户测试，因而暂时无法准确证明：

- 视觉选择是否真正缩短了需求确认时间；
- 七轮选择是否过长；
- 客户是否认可系统生成的画像；
- 推荐理由是否能够帮助客户决策；
- 矛盾提醒是否自然且容易接受；
- 系统是否真正减少设计师改图次数；
- 系统对签单率、交付效率和客户满意度的提升程度。

### 2.11 本地存储不等同于正式持久化系统

当前多客户数据保存在访问者浏览器的 `localStorage` 中。这种方式使GitHub下载者无需配置数据库即可运行完整MVP，也避免比赛版采集不必要的敏感信息，但存在明确边界：

- 数据只存在于当前电脑、当前浏览器和当前浏览器配置文件中；
- 不同设备、不同浏览器及不同员工之间不能共享客户数据；
- 清理站点数据、更换浏览器或重装系统后，自建客户数据可能丢失；
- 没有服务端备份、恢复、版本记录和操作审计；
- 无法支持销售、设计师和店长围绕同一客户协同；
- 不适合保存正式客户资料、合同、报价、订单或其他敏感业务数据。

因此，当前本地存储方案适合比赛展示和离线体验，但不能替代正式数据库、身份认证和企业数据治理体系。

### 2.12 企业级安全与权限体系尚未完善

系统已经明确案例需要授权、脱敏和标注，但正式使用还需要完善：

- 客户身份认证及员工账号体系；
- 客户、销售、设计师、店长和总部的分级权限；
- 敏感数据加密、操作日志和审计记录；
- 案例授权、撤回和删除机制；
- AI训练数据的使用范围控制；
- 图片、设计方案和第三方素材的版权管理；
- 数据备份、容灾、保留期限和彻底删除机制。

## 3. 后续完善方向

### 3.1 建设真正的AI家装顾问Agent

下一阶段应在现有视觉选择器、画像引擎、推荐算法和交付看板之上增加对话式Agent，将自然语言沟通与图片选择结合起来。

Agent不替代现有模块，而是作为统一入口，根据客户意图调用不同工具：

```text
客户自然语言对话
        ↓
AI家装顾问Agent
        ├── 客户画像读取与更新
        ├── 视觉偏好选择器
        ├── 案例检索工具
        ├── 推荐算法
        ├── 预算与需求矛盾检查
        ├── SKU与交付查询
        └── 转交设计师或销售人员
```

Agent应具备以下核心能力：

1. 根据已知信息主动提出下一条最有价值的问题；
2. 将多轮对话转换为带来源和置信度的结构化画像；
3. 在文字难以确认偏好时主动调用视觉选择器；
4. 允许客户通过自然语言更新、撤销或补充画像；
5. 调用推荐引擎并解释每套方案的匹配依据和取舍；
6. 识别需求矛盾并与客户共同确认折中方案；
7. 仅依据工具返回的数据回答价格、库存和交期问题；
8. 在低置信度或高风险场景下主动转交人工。

Agent建设可以分为三个版本：

- **Agent V1：需求访谈助手。** 实现多轮对话、上下文记忆、画像字段提取、信息缺失追问和视觉选择器调用。
- **Agent V2：方案推荐助手。** 接入案例检索、推荐算法和矛盾检查，支持对话式方案比较与局部修改。
- **Agent V3：全流程客户管家。** 接入真实SKU、报价、订单和交付系统，支持进度查询、风险解释、售后报修与人工转接。

### 3.2 开展真实用户验证与体验优化

近期应优先验证MVP是否真正解决客户需求表达问题。建议邀请10—30名潜在装修用户，对比传统文字沟通与AI视觉选择两种方式，记录：

- 完成需求确认所需时间；
- 明确需求项数量；
- 视觉选择流程完成率；
- 首次推荐满意度；
- 返回、撤销、跳过和修改次数；
- 客户对最终画像的认可度；
- 是否愿意继续咨询设计师或销售人员。

测试结果应用于调整选择轮次、图片数量、页面文案、矛盾提醒方式和推荐解释内容。

### 3.3 建立可持续的案例管理后台

后续应增加案例管理后台，实现：

- 新增、编辑、审核、下架和批量导入案例；
- 自动识别图片风格、颜色、材质和空间类型；
- 由设计师确认或修改AI标签；
- 管理案例来源、客户授权、脱敏状态和图片版权；
- 明确区分官网参考、AI生成、模拟案例和真实成交案例；
- 统计案例浏览量、收藏率、推荐率、修改率和转化率；
- 识别重复、低质量和过时案例。

未来应逐步使用真实、授权、脱敏的成交案例替换模拟数据。

### 3.4 升级自适应视觉选择器

将固定七轮问答升级为动态流程：

1. AI分析客户的自然语言描述；
2. 判断哪些偏好已经明确；
3. 只针对不确定项提供图片选择；
4. 根据每次选择计算偏好置信度；
5. 置信度达到要求后自动结束该类别；
6. 发现矛盾时插入针对性确认；
7. 为不同客户生成不同的选择路径。

### 3.5 建立家庭生活方式画像

在现有审美偏好基础上，进一步建立四层画像：

| 画像层级 | 主要内容 |
| --- | --- |
| 基础画像 | 地区、户型、面积、预算、入住时间 |
| 审美画像 | 风格、色彩、材质、灯光和个性元素 |
| 功能画像 | 收纳、动线、空间功能和智能家居需求 |
| 生活画像 | 家庭成员、行为习惯、安全需求和未来变化 |

### 3.6 接入户型图和真实空间理解

支持上传户型图、量房图或现场照片，并使用多模态AI识别：

- 房间边界、尺寸和面积；
- 门窗、梁柱和承重结构；
- 厨卫水电条件；
- 采光、朝向和可布置墙面；
- 空间动线和潜在碰撞；
- 柜体安装条件。

推荐算法应先判断方案是否适合真实户型，再进行风格和视觉匹配。

### 3.7 升级混合推荐算法

在现有规则推荐基础上，逐步形成以下推荐架构：

```text
硬性条件过滤
→ 规则匹配评分
→ 图片与语义向量检索
→ 相似客户行为推荐
→ 成交和满意度排序
→ 风险与可交付性校正
→ 推荐理由生成
```

预算、户型、安全和产品状态应作为硬性条件；风格、材质和色彩作为偏好条件；成交率、满意度、修改次数和售后情况作为质量信号。

### 3.8 打通真实SKU、报价与供应链

在获得企业授权后，逐步连接产品数据库、门店报价系统、库存与生产系统、物流安装系统及CRM，让效果图中的主要产品都能够回答：

> 是什么产品、多少钱、能否生产、何时交付、由谁安装。

同时建立效果图、产品清单、报价单、合同和订单之间的一致性检查，减少漏项、错项和错误承诺。

### 3.9 建立可控的AI效果图生成能力

效果图生成应采用“案例检索＋户型约束＋真实SKU＋局部生成”的技术路径：

- 检索与客户画像相似的优质案例；
- 保留真实户型结构和关键尺寸；
- 优先使用真实在售SKU和标准材质；
- 生成不同预算档位的候选方案；
- 支持局部更换柜门、颜色、灯光和软装；
- 自动检查不可生产或存在安全风险的结构；
- 持续保留清晰的“AI生成示意图”标识；
- 由设计师完成专业审核后才能用于正式方案。

### 3.10 完善交付预测与风险闭环

将交付看板升级为真实业务事件驱动系统，并增加：

- 订单延期概率预测；
- 跨品类齐套风险识别；
- 缺件、错件、返工和报价漏项检测；
- 安装资源冲突提醒；
- 风险等级、责任人和处理时限；
- 超时升级、客户通知和处理复核；
- 风险关闭及复盘记录。

### 3.11 建立数据闭环和持续优化机制

系统应在获得客户授权的前提下记录完整业务链路：

```text
客户描述
→ 图片浏览与选择
→ 客户画像
→ 案例推荐
→ 方案修改
→ 报价与签约
→ 生产交付
→ 售后处理
→ 客户满意度
```

这些数据可以用于校准推荐权重、识别高修改率设计、发现区域偏好，并反向支持产品研发、门店样板间布置和营销素材选择。

### 3.12 从本地存储升级为受控持久化与协同

在MVP通过用户验证并获得企业授权后，可逐步将浏览器本地存储升级为服务端持久化方案。升级不应只是“增加一个数据库”，而应同时解决身份、权限、数据生命周期和协同问题：

- 为客户、销售、设计师、店长和总部建立分级身份与权限；
- 将客户需求、视觉选择、画像、推荐记录和处置日志保存为可追踪版本；
- 支持跨设备恢复、多人协同、导出、备份和授权删除；
- 对敏感字段进行最小化采集、加密和访问审计；
- 为正式业务数据设置保留期限、撤回授权和彻底删除机制；
- 保留“纯本地演示模式”，确保比赛和公开仓库仍可在不连接企业数据的情况下运行。

在获得真实客户和企业数据之前，不应把模拟数据自动上传到公共云端，也不应把本地演示数据描述为欧派真实CRM数据。

## 6. 建议验收标准

### 6.1 当前版本已经完成的验证基线

- 可以在同一浏览器创建、编辑、删除和切换多名匿名客户；
- 每名客户的需求、视觉选择、画像、推荐和交付处置状态相互隔离；
- 刷新页面后能够从 `localStorage` 恢复客户列表和各自进度；
- “林女士”演示客户可以一键载入和恢复；
- 页面和本地存储结构不包含手机号及详细地址字段；
- 桌面端与390像素宽移动端可以完成主要流程且无横向溢出；
- 项目根目录可以直接启动，默认通过 `http://localhost:3000` 访问；
- 以上能力不依赖PostgreSQL或远程数据库。

这些结果属于本地MVP的技术验证，不代表已经完成真实门店部署、生产环境压测或企业级数据合规验收。

### 6.2 对话Agent验收标准

- 能连续完成不少于10轮上下文相关对话；
- 能从自然语言中提取户型、预算、家庭结构和核心偏好；
- 信息不足时能够主动追问，而不是直接编造结论；
- 能主动调用视觉选择器，而不是只返回文字；
- 能通过对话更新、撤销和补充画像字段；
- 能识别不少于5类需求矛盾；
- 能调用推荐引擎返回并解释三套方案；
- 推荐结论能够说明数据来源和匹配理由；
- 不编造价格、库存、SKU和交期；
- 低置信度或高风险问题能够转交人工；
- 在客户授权范围内能够恢复历史对话和客户画像。

### 6.3 产品与用户体验验收标准

- 视觉选择流程完成率不低于80%；
- 平均需求确认时间控制在10分钟以内；
- 客户对最终画像的认可度不低于80%；
- 首次推荐满意度不低于75%；
- 需求或方案修改次数较传统方式降低20%以上；
- 所有案例均具有明确的数据来源和图片属性标识；
- 所有AI图片均在缩略图、放大图和演示截图中清楚标注。

以上指标属于下一阶段的验证目标，不代表当前系统已经取得的实际业务成果。

## 7. 总结

当前项目最大的优势，是已经完成了一条可运行、可演示、可解释的客户体验主链路，并进一步摆脱了固定客户演示数据，支持完整需求输入、多名匿名客户切换以及浏览器本地保存恢复。最大的不足，仍是缺少真正的对话式AI Agent、真实企业数据、实际业务系统连接和足够规模的用户验证；同时，本地存储还不能承担跨设备协同、服务端备份和企业级数据治理职责。

后续建设重点应从单纯增加演示功能转向以下三个方向：

> 用对话式Agent降低客户表达难度，用真实测试和真实数据提升可信度，用业务系统连接保证方案能够报价、生产和交付。

当系统能够让客户像与专业家装顾问聊天一样表达需求，并由Agent主动追问、调用视觉选择、更新客户画像、解释推荐结果、查询真实业务数据并适时转交人工时，“欧派AI灵感家”才能从比赛MVP进一步发展为具有真实应用价值的大家居客户体验平台。
